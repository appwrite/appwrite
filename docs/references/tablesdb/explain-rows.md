Get the query plan for a `listRows` call. Useful for diagnosing slow reads and verifying that the indexes you created on a table are actually being used.

Takes the same parameters as `listRows`. The query is executed so the plan can report real execution stats (`rowsReturned`, `executionTime`) alongside the planner estimates (`rowsScanned`, `indexUsed`, `estimatedCost`); the rows themselves are discarded and never returned. Returns one plan entry per physical query Appwrite runs — including the `count` for the total and the per-relationship fetches that `listRows` issues sequentially when your query selects related fields. The metrics are engine-agnostic, and internal storage details (the backing engine, the permission companion table, the metadata system table, internal column names) are not exposed.
