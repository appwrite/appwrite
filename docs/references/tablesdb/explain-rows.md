Get the query plan for a `listRows` call. Useful for diagnosing slow reads and verifying that the indexes you created on a table are actually being used.

Supports the same `queries` and `total` behavior as `listRows`. The query is executed so the plan can report stable Appwrite-level `metrics` (`estimatedRecordsScanned`, `recordsReturned`, `durationMs`, `estimatedCost`) and an `access` summary (`type`, `index`); the rows themselves are discarded and never returned. Returns one plan entry per physical query Appwrite runs — including the `count` for the total and the per-relationship fetches that `listRows` issues sequentially when your query selects related fields. The response also includes a sanitized, backend-specific `tree` for advanced inspection, with internal storage identifiers stripped and the backing engine not identified.
