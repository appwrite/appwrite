Get the query plan for a `listDocuments` call. Useful for diagnosing slow reads and verifying that the indexes you created on a collection are actually being used.

Takes the same parameters as `listDocuments`. The query is executed so the plan can report real execution stats (`rowsReturned`, `executionTime`) alongside the planner estimates (`rowsScanned`, `indexUsed`, `estimatedCost`); the documents themselves are discarded and never returned. Returns one plan entry per physical query Appwrite runs — including the `count` for the total and the per-relationship fetches that `listDocuments` issues sequentially when your query selects related fields. The metrics are engine-agnostic; the response also includes a sanitized `tree` with the raw engine plan (scan type, candidate indexes, filter conditions) for deep diagnosis, with internal storage identifiers (permission and metadata tables, internal column names) stripped and the backing engine not identified.
