Get the query plan for a `listDocuments` call. Useful for diagnosing slow reads and verifying that the indexes you created on a collection are actually being used.

Supports the same `queries` and `total` behavior as `listDocuments`. The query is executed so the plan can report stable Appwrite-level `metrics` (`estimatedRecordsScanned`, `recordsReturned`, `durationMs`, `estimatedCost`) and an `access` summary (`type`, `index`); the documents themselves are discarded and never returned. Returns one plan entry per physical query Appwrite runs — including the `count` for the total and the per-relationship fetches that `listDocuments` issues sequentially when your query selects related fields. The response also includes a sanitized, backend-specific `tree` for advanced inspection, with internal storage identifiers and runtime metadata stripped. The backing database is not identified.
