Get the query plan for a `listDocuments` call. Useful for diagnosing slow vector searches and verifying that the indexes (HNSW, IVFFLAT, etc.) you created on a collection are actually being used.

Takes the same parameters as `listDocuments`. The query is executed so the plan can report real execution stats (`rowsReturned`, `executionTime`) alongside the planner estimates (`engine`, `rowsScanned`, `indexUsed`, `estimatedCost`) and the raw engine plan `tree`; the documents themselves are discarded and never returned. Returns one plan entry per physical query Appwrite runs — including the `count` for the total and the per-relationship fetches that `listDocuments` issues sequentially when your query selects related fields. Internal storage details (the permission companion table, the metadata system table, internal column names) are stripped from the output, so the response only references your collections and `$`-prefixed attributes.
